🧪 Tool Persistence 功能测试

📋 测试说明

本页面用于测试AI诊断页面的skill调用信息持久化功能。

测试目标：验证切换会话后，skill调用信息能够正确恢复显示。

🔧 功能实现

后端改造

  • ✅ 在WebSocket中保存tool_call消息到数据库（role=tool_call）
  • ✅ 在WebSocket中保存tool_result消息到数据库（role=tool_result）
  • ✅ 消息内容使用JSON格式存储tool_name、tool_args、result等信息

前端改造

  • ✅ 扩展loadMessages()方法，处理tool_call和tool_result消息
  • ✅ 新增_restoreToolCall()方法，恢复tool调用显示
  • ✅ 新增_restoreToolResult()方法，恢复tool执行结果
  • ✅ 使用restoredTools Map匹配tool_call和tool_result

🧪 测试步骤

  1. 步骤1：打开AI诊断页面，创建新会话
    http://localhost:9939/#/diagnosis
  2. 步骤2：选择一个数据源，发送一个会触发skill调用的问题
    例如："查询慢查询" 或 "分析数据库性能"
  3. 步骤3：观察右侧Tool Panel，确认skill调用信息正常显示
    应该看到：
    • Skill名称
    • Arguments（参数）
    • Result（结果）
    • 执行时间
  4. 步骤4：创建另一个新会话，切换到新会话
    点击左侧"New"按钮
  5. 步骤5：切换回第一个会话
    点击左侧会话列表中的第一个会话
  6. 步骤6：验证Tool Panel中的skill调用信息是否恢复显示
    预期结果：
    • ✅ 对话消息正常显示
    • ✅ Tool Panel中的skill调用信息完整恢复
    • ✅ Skill状态显示正确（Complete/Error）
    • ✅ 执行时间正确显示

📊 数据库验证

可以通过以下SQL查询验证数据是否正确保存：

-- 查询某个会话的所有消息（包括tool消息） SELECT id, session_id, role, CASE WHEN role IN ('tool_call', 'tool_result') THEN content::json->>'tool_name' ELSE LEFT(content, 50) END as content_preview, created_at FROM chat_messages WHERE session_id = 1 -- 替换为实际的session_id ORDER BY created_at; -- 统计各类型消息数量 SELECT role, COUNT(*) as count FROM chat_messages WHERE session_id = 1 -- 替换为实际的session_id GROUP BY role;
