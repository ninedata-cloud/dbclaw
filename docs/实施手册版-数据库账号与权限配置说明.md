# 实施手册版：数据库账号与权限配置说明

本文面向实施工程师、DBA、安全管理员，用于指导 DBClaw 接入数据库时创建账号、授予权限、解释权限用途，并对可选高权限能力进行边界说明。

适用版本：基于当前 DBClaw 产品代码能力整理
整理时间：2026-04-23

## 一、目标与原则

DBClaw 是数据库智能运维平台，默认需要读取数据库的以下信息：

- 业务数据：用于只读诊断、执行计划、表统计、索引统计
- 元数据：schema、table、column、index、view、parameter
- 系统状态：连接数、活动会话、锁等待、数据库大小、启动时间
- 性能数据：慢 SQL、TOP SQL、等待事件、缓存命中率、复制状态

因此，本手册采用统一原则：

- 账号必须是专用监控账号
- 默认是只读权限
- 默认可读取所有业务数据与系统监控数据
- 默认不允许修改业务数据和结构
- 默认不允许终止会话

## 二、权限分层

### 1. 基础只读监控权限

包含：

- 连接数据库
- 读取业务对象
- 读取系统视图与字典
- 执行只读 SQL
- 执行 `EXPLAIN` 或执行计划查询

### 2. 可选增强权限

仅在客户明确需要时追加：

- 取消查询
- 终止会话
- 审计日志读取
- 额外的 TOP SQL 扩展视图权限

### 3. 不建议默认授予的权限

不建议默认授予：

- `INSERT`
- `UPDATE`
- `DELETE`
- `CREATE`
- `ALTER`
- `DROP`
- `GRANT`
- `REVOKE`
- 超级管理员权限

## 三、产品实际依赖说明

基于当前 DBClaw 代码，数据库接入后会用到以下典型能力：

- PostgreSQL / openGauss：`pg_stat_activity`、`pg_stat_database`、`pg_settings`、`pg_stat_user_tables`、`pg_stat_statements`
- MySQL：`information_schema`、`performance_schema`、`SHOW GLOBAL STATUS`、`SHOW GLOBAL VARIABLES`、`PROCESSLIST`
- SQL Server：DMV/DMF、`sys.dm_exec_sessions`、`sys.dm_exec_requests`、`sys.dm_exec_query_stats`、`sys.configurations`
- Oracle：`V$SESSION`、`V$SQL`、`V$SYSSTAT`、`V$PARAMETER`、`DBA_*`
- SAP HANA：`SYS.M_*`、`SYS.TABLES`、`SYS.TABLE_COLUMNS`、`SYS.INDEXES`

因此，最简单的落地思路不是逐条对象授权，而是优先采用数据库内置的只读监控角色或统一全局只读权限。

## 四、各数据库实施脚本与说明

## PostgreSQL

### 推荐实施方式

推荐直接使用预定义角色：

- `pg_monitor`
- `pg_read_all_data`

再补所有数据库的 `CONNECT`。

### 推荐脚本

```sql
CREATE ROLE dbclaw_mon
LOGIN
PASSWORD '<强密码>';

GRANT pg_monitor TO dbclaw_mon;
GRANT pg_read_all_data TO dbclaw_mon;

SELECT format('GRANT CONNECT ON DATABASE %I TO dbclaw_mon;', datname)
FROM pg_database
WHERE datistemplate = false
  AND datallowconn = true
\gexec
```

### 权限说明

- `pg_monitor`：读取系统统计信息、监控视图、配置等
- `pg_read_all_data`：读取所有 schema 中的表、视图、序列
- `CONNECT`：允许连接到每个数据库

### 可选增强

终止会话：

```sql
GRANT pg_signal_backend TO dbclaw_mon;
```

TOP SQL：

```sql
CREATE EXTENSION IF NOT EXISTS pg_stat_statements;
GRANT SELECT ON pg_stat_statements TO dbclaw_mon;
```

### 实施备注

- 这是 PostgreSQL 最推荐的统一方案
- 如果现场启用了 RLS，业务表访问仍受 RLS 限制
- 若实例默认已经允许 `PUBLIC` 连接数据库，则 `CONNECT` 可能不是必需，但建议显式授予

## openGauss

### 推荐实施方式

openGauss 版本差异相对较大，实施建议分两步：

1. 先尝试 PostgreSQL 兼容预定义角色方案
2. 如果角色不存在，再改显式授权方案

### 方案 A：兼容 PostgreSQL 角色

```sql
CREATE USER dbclaw_mon IDENTIFIED BY '<强密码>';

GRANT pg_monitor TO dbclaw_mon;
GRANT pg_read_all_data TO dbclaw_mon;

GRANT CONNECT ON DATABASE postgres TO dbclaw_mon;
```

### 方案 B：显式授权

```sql
CREATE USER dbclaw_mon IDENTIFIED BY '<强密码>';

GRANT CONNECT ON DATABASE appdb TO dbclaw_mon;
```

切到业务库后执行：

```sql
GRANT USAGE ON SCHEMA public TO dbclaw_mon;
GRANT SELECT ON ALL TABLES IN SCHEMA public TO dbclaw_mon;
GRANT SELECT ON ALL SEQUENCES IN SCHEMA public TO dbclaw_mon;

ALTER DEFAULT PRIVILEGES IN SCHEMA public
GRANT SELECT ON TABLES TO dbclaw_mon;

ALTER DEFAULT PRIVILEGES IN SCHEMA public
GRANT SELECT ON SEQUENCES TO dbclaw_mon;

GRANT SELECT ON pg_catalog.pg_stat_activity TO dbclaw_mon;
GRANT SELECT ON pg_catalog.pg_stat_database TO dbclaw_mon;
GRANT SELECT ON pg_catalog.pg_database TO dbclaw_mon;
GRANT SELECT ON pg_catalog.pg_settings TO dbclaw_mon;
GRANT SELECT ON pg_catalog.pg_stat_user_tables TO dbclaw_mon;
GRANT SELECT ON pg_catalog.pg_stat_user_indexes TO dbclaw_mon;
GRANT SELECT ON pg_catalog.pg_stat_replication TO dbclaw_mon;
```

### 可选增强

如已安装 `pg_stat_statements`：

```sql
GRANT SELECT ON pg_catalog.pg_stat_statements TO dbclaw_mon;
```

### 实施备注

- 现场若有多个业务库，应对每个库单独做 `CONNECT`
- 现场若有多个业务 schema，应对每个 schema 单独做 `USAGE + SELECT`

## MySQL / TDSQL-C MySQL

### 推荐实施方式

推荐采用全实例只读监控账号，脚本最短，也最便于客户审批。

### 推荐脚本

```sql
CREATE USER IF NOT EXISTS 'dbclaw_mon'@'%'
IDENTIFIED BY '<强密码>';

GRANT SELECT, SHOW VIEW, PROCESS, REPLICATION CLIENT
ON *.* TO 'dbclaw_mon'@'%';

FLUSH PRIVILEGES;
```

### 权限说明

- `SELECT`：读取所有数据库中的表与视图
- `SHOW VIEW`：读取视图定义
- `PROCESS`：读取所有连接、线程、会话信息
- `REPLICATION CLIENT`：读取复制状态

### 可选增强

终止会话：

```sql
GRANT CONNECTION_ADMIN ON *.* TO 'dbclaw_mon'@'%';
FLUSH PRIVILEGES;
```

### 实施备注

- 如果客户只允许访问特定业务库，也可以把 `SELECT, SHOW VIEW ON *.*` 改成指定库
- 但 DBClaw 当前定位是统一监控诊断平台，通常建议统一实例只读
- 若要使用 TOP SQL，数据库需启用 `performance_schema`

## SQL Server

### 推荐实施方式

SQL Server 2014+ 推荐使用服务器级授权，避免逐库配置复杂度。

### 推荐脚本

```sql
USE [master];
GO

CREATE LOGIN [dbclaw_mon]
WITH PASSWORD = '<强密码>',
     CHECK_POLICY = ON,
     CHECK_EXPIRATION = ON;
GO

GRANT VIEW ANY DATABASE TO [dbclaw_mon];
GRANT CONNECT ANY DATABASE TO [dbclaw_mon];
GRANT SELECT ALL USER SECURABLES TO [dbclaw_mon];
GRANT VIEW ANY DEFINITION TO [dbclaw_mon];
GO

IF CAST(SERVERPROPERTY('ProductMajorVersion') AS int) >= 16
    GRANT VIEW SERVER PERFORMANCE STATE TO [dbclaw_mon];
ELSE
    GRANT VIEW SERVER STATE TO [dbclaw_mon];
GO
```

### 权限说明

- `VIEW ANY DATABASE`：可见所有数据库
- `CONNECT ANY DATABASE`：可连接所有数据库
- `SELECT ALL USER SECURABLES`：可读取所有用户对象
- `VIEW ANY DEFINITION`：可读取对象定义与元数据
- `VIEW SERVER STATE` / `VIEW SERVER PERFORMANCE STATE`：读取服务器级 DMV/性能状态

### 执行计划补充脚本

如需执行计划能力，补数据库级 `SHOWPLAN`：

```sql
DECLARE @sql nvarchar(max) = N'';

SELECT @sql += N'
USE ' + QUOTENAME(name) + N';
IF USER_ID(N''dbclaw_mon'') IS NULL
    CREATE USER [dbclaw_mon] FOR LOGIN [dbclaw_mon];
GRANT SHOWPLAN TO [dbclaw_mon];
'
FROM sys.databases
WHERE database_id > 4
  AND state_desc = 'ONLINE';

EXEC sys.sp_executesql @sql;
GO
```

### 可选增强

终止会话：

```sql
GRANT ALTER ANY CONNECTION TO [dbclaw_mon];
GO
```

### 实施备注

- SQL Server 2012 及更老版本，推荐改成逐库 `db_datareader + SHOWPLAN`
- SQL Server 2022 开始，部分 DMV 推荐使用 `VIEW SERVER PERFORMANCE STATE`

## Oracle

### 推荐实施方式

Oracle 推荐采用简单统一模型：

- `CREATE SESSION`
- `SELECT_CATALOG_ROLE`
- `SELECT ANY TABLE`
- `EXECUTE ON SYS.DBMS_XPLAN`

### 推荐脚本

```sql
CREATE USER DBCLAW_MON
IDENTIFIED BY "<强密码>"
DEFAULT TABLESPACE USERS
TEMPORARY TABLESPACE TEMP
ACCOUNT UNLOCK;

GRANT CREATE SESSION TO DBCLAW_MON;
GRANT SELECT_CATALOG_ROLE TO DBCLAW_MON;
GRANT SELECT ANY TABLE TO DBCLAW_MON;
GRANT EXECUTE ON SYS.DBMS_XPLAN TO DBCLAW_MON;
```

### 权限说明

- `CREATE SESSION`：允许登录
- `SELECT ANY TABLE`：读取所有业务表
- `SELECT_CATALOG_ROLE`：读取数据字典、`DBA_*`、`V$*`
- `DBMS_XPLAN`：支持执行计划展示

### 可选增强

终止会话：

```sql
GRANT ALTER SYSTEM TO DBCLAW_MON;
```

### 实施备注

- 建议在目标 PDB 中创建监控账号
- 如客户对 `SELECT ANY TABLE` 敏感，也可改成按 schema 批量授权，但实施与维护成本会显著提升

## SAP HANA

### 推荐实施方式

建议创建自定义角色，统一封装授权。

### 推荐脚本

```sql
CREATE USER DBCLAW_MON PASSWORD "<强密码>" NO FORCE_FIRST_PASSWORD_CHANGE;

CREATE ROLE DBCLAW_MONITOR_ROLE;

GRANT MONITORING TO DBCLAW_MONITOR_ROLE;
GRANT CATALOG READ TO DBCLAW_MONITOR_ROLE;

GRANT SELECT ON SCHEMA "APP_SCHEMA" TO DBCLAW_MONITOR_ROLE;
GRANT SELECT ON SCHEMA "APP_SCHEMA2" TO DBCLAW_MONITOR_ROLE;

GRANT DBCLAW_MONITOR_ROLE TO DBCLAW_MON;
```

### 权限说明

- `MONITORING`：读取监控视图与系统状态
- `CATALOG READ`：读取目录与元数据
- `SELECT ON SCHEMA`：读取业务 schema 中的数据

### 可选增强

终止会话：

```sql
GRANT SESSION ADMIN TO DBCLAW_MON;
```

### 实施备注

- 若业务 schema 较多，建议统一梳理后集中授权
- 若客户希望最小权限模型，可把角色进一步拆分，但不建议作为第一版实施方案

## 五、DBClaw 自身元数据库账号

DBClaw 自己需要一个 PostgreSQL 元数据库账号，用于保存平台配置、数据源、告警、巡检结果等平台数据。

推荐单独创建，不与业务监控账号混用。

### 推荐脚本

```sql
CREATE ROLE dbclaw_meta
LOGIN
PASSWORD '<强密码>';

CREATE DATABASE dbclaw
  OWNER dbclaw_meta
  ENCODING 'UTF8';
```

### 连接串示例

```env
DATABASE_URL=postgresql+asyncpg://dbclaw_meta:<强密码>@127.0.0.1:5432/dbclaw
```

## 六、实施检查清单

实施完成后，建议逐项确认：

- 账号可以正常登录
- DBClaw 中数据源连接测试通过
- 可以读取库状态、连接数、参数、库大小
- 可以读取业务表统计与索引统计
- 可以执行只读诊断 SQL
- 可以执行执行计划查询
- 若客户启用了 TOP SQL，可正常读取对应扩展视图
- 若客户未申请高权限，确认账号无法写入业务数据、无法终止会话

## 七、推荐交付口径

建议在客户交付与安全审批中使用以下描述：

- 监控账号为专用只读账号
- 默认可读取所有业务数据、元数据和系统监控数据
- 默认不包含写权限
- 默认不包含终止会话权限
- 高危权限需单独审批、单独授权

## 八、版本与兼容性说明

- PostgreSQL / openGauss：预定义角色与扩展能力因版本不同可能存在差异
- MySQL / TDSQL-C MySQL：动态权限与 `performance_schema` 能力受版本影响
- SQL Server：2014+ 适合采用服务器级统一授权；2012 及以下建议逐库处理
- Oracle：建议在目标 PDB 中实施
- SAP HANA：不同版本对监控角色与系统视图可见性可能略有差异
