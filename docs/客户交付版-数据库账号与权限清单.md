# 客户交付版：数据库账号与权限清单

本文面向客户、售前、实施与安全审批人员，说明 DBClaw 接入数据库时建议使用的账号类型、最简授权方式与可选增强权限。

适用版本：基于当前 DBClaw 产品代码能力整理
整理时间：2026-04-23

## 一、总体原则

- 建议为每个数据库实例单独创建一个专用监控账号，不使用 `root`、`sys`、`sa`、`sysdba` 等超级账号。
- 默认权限模型为“读取所有业务数据 + 读取所有元数据/系统监控数据”。
- 默认不授予 DDL、DML、DCL 权限，不允许修改业务数据或结构。
- 默认不授予“终止会话”权限，如客户确有需要，再单独申请。
- 对于同一实例下多个业务库，可复用同一个监控账号，但仍建议在 DBClaw 中按库分别配置数据源。

## 二、权限口径

### 1. 默认权限范围

默认需要具备以下能力：

- 读取业务表、视图、序列等业务对象
- 读取系统视图、数据字典、性能视图、连接信息、锁等待信息
- 读取数据库参数、库大小、表统计、索引统计、慢 SQL / TOP SQL 所依赖的数据
- 执行只读诊断 SQL 与执行计划查询

### 2. 默认不包含的权限

默认不包含以下能力：

- 新增、修改、删除业务数据
- 创建、删除、修改数据库对象
- 修改用户与权限
- 终止数据库会话
- 执行任意写入类 SQL

### 3. 可选增强权限

如客户明确需要，可单独加：

- 终止会话 / 取消查询
- 更高阶的 TOP SQL、扩展视图或审计日志读取权限

## 三、各数据库建议脚本

## PostgreSQL

### 推荐说明

- 最简单方案：`pg_monitor + pg_read_all_data + CONNECT`
- `pg_monitor` 负责系统监控与状态视图
- `pg_read_all_data` 负责读取所有业务对象
- 对每个数据库还需要 `CONNECT`

### 最简脚本

```sql
CREATE ROLE dbclaw_mon
LOGIN
PASSWORD '<强密码>';

GRANT pg_monitor TO dbclaw_mon;
GRANT pg_read_all_data TO dbclaw_mon;

SELECT format('GRANT CONNECT ON DATABASE %I TO dbclaw_mon;', datname)
FROM pg_database
WHERE datistemplate = false
  AND datallowconn = true
\gexec
```

### 可选增强

如需终止会话：

```sql
GRANT pg_signal_backend TO dbclaw_mon;
```

如需 TOP SQL：

```sql
CREATE EXTENSION IF NOT EXISTS pg_stat_statements;
GRANT SELECT ON pg_stat_statements TO dbclaw_mon;
```

### 一句话说明

一个账号即可覆盖实例监控与全库只读诊断，是 PostgreSQL 最推荐的授权方式。

## openGauss

### 推荐说明

- 优先按 PostgreSQL 兼容方式授权
- 若现场版本不支持预定义角色，再改为显式授权

### 方案 A：兼容 PostgreSQL 角色时

```sql
CREATE USER dbclaw_mon IDENTIFIED BY '<强密码>';

GRANT pg_monitor TO dbclaw_mon;
GRANT pg_read_all_data TO dbclaw_mon;

GRANT CONNECT ON DATABASE postgres TO dbclaw_mon;
```

如有多个数据库，对每个数据库重复执行一次 `GRANT CONNECT ON DATABASE <库名>`。

### 方案 B：不支持预定义角色时

```sql
CREATE USER dbclaw_mon IDENTIFIED BY '<强密码>';

GRANT CONNECT ON DATABASE appdb TO dbclaw_mon;
```

切到业务库后执行：

```sql
GRANT USAGE ON SCHEMA public TO dbclaw_mon;
GRANT SELECT ON ALL TABLES IN SCHEMA public TO dbclaw_mon;
GRANT SELECT ON ALL SEQUENCES IN SCHEMA public TO dbclaw_mon;

ALTER DEFAULT PRIVILEGES IN SCHEMA public
GRANT SELECT ON TABLES TO dbclaw_mon;

ALTER DEFAULT PRIVILEGES IN SCHEMA public
GRANT SELECT ON SEQUENCES TO dbclaw_mon;

GRANT SELECT ON pg_catalog.pg_stat_activity TO dbclaw_mon;
GRANT SELECT ON pg_catalog.pg_stat_database TO dbclaw_mon;
GRANT SELECT ON pg_catalog.pg_database TO dbclaw_mon;
GRANT SELECT ON pg_catalog.pg_settings TO dbclaw_mon;
GRANT SELECT ON pg_catalog.pg_stat_user_tables TO dbclaw_mon;
GRANT SELECT ON pg_catalog.pg_stat_user_indexes TO dbclaw_mon;
GRANT SELECT ON pg_catalog.pg_stat_replication TO dbclaw_mon;
```

### 一句话说明

openGauss 建议优先走 PostgreSQL 兼容授权；若版本差异较大，再退回显式视图授权。

## MySQL / TDSQL-C MySQL

### 推荐说明

- 最简单方案：全实例只读 + 监控管理只读
- `SELECT, SHOW VIEW` 负责读取所有业务库与视图
- `PROCESS` 负责查看连接与线程
- `REPLICATION CLIENT` 负责查看复制状态

### 最简脚本

```sql
CREATE USER IF NOT EXISTS 'dbclaw_mon'@'%'
IDENTIFIED BY '<强密码>';

GRANT SELECT, SHOW VIEW, PROCESS, REPLICATION CLIENT
ON *.* TO 'dbclaw_mon'@'%';

FLUSH PRIVILEGES;
```

### 可选增强

如需终止会话：

```sql
GRANT CONNECTION_ADMIN ON *.* TO 'dbclaw_mon'@'%';
FLUSH PRIVILEGES;
```

### 一句话说明

一个全局只读监控账号即可覆盖大多数 MySQL 监控与诊断场景，最容易实施与解释。

## SQL Server

### 推荐说明

- SQL Server 2014+ 推荐使用服务器级授权
- 可一次性实现“能连所有库 + 能读所有库 + 能看系统状态”
- 逻辑比逐库建用户更简单

### 最简脚本

```sql
USE [master];
GO

CREATE LOGIN [dbclaw_mon]
WITH PASSWORD = '<强密码>',
     CHECK_POLICY = ON,
     CHECK_EXPIRATION = ON;
GO

GRANT VIEW ANY DATABASE TO [dbclaw_mon];
GRANT CONNECT ANY DATABASE TO [dbclaw_mon];
GRANT SELECT ALL USER SECURABLES TO [dbclaw_mon];
GRANT VIEW ANY DEFINITION TO [dbclaw_mon];
GO

IF CAST(SERVERPROPERTY('ProductMajorVersion') AS int) >= 16
    GRANT VIEW SERVER PERFORMANCE STATE TO [dbclaw_mon];
ELSE
    GRANT VIEW SERVER STATE TO [dbclaw_mon];
GO
```

### 如需执行计划

```sql
DECLARE @sql nvarchar(max) = N'';

SELECT @sql += N'
USE ' + QUOTENAME(name) + N';
IF USER_ID(N''dbclaw_mon'') IS NULL
    CREATE USER [dbclaw_mon] FOR LOGIN [dbclaw_mon];
GRANT SHOWPLAN TO [dbclaw_mon];
'
FROM sys.databases
WHERE database_id > 4
  AND state_desc = 'ONLINE';

EXEC sys.sp_executesql @sql;
GO
```

### 可选增强

如需终止会话：

```sql
GRANT ALTER ANY CONNECTION TO [dbclaw_mon];
GO
```

### 一句话说明

SQL Server 最简单的方式是服务器级统一只读授权，再按需补 `SHOWPLAN`。

## Oracle

### 推荐说明

- 最简单方案：`CREATE SESSION + SELECT_CATALOG_ROLE + SELECT ANY TABLE`
- `SELECT ANY TABLE` 负责读所有业务数据
- `SELECT_CATALOG_ROLE` 负责读数据字典、系统视图与性能视图

### 最简脚本

```sql
CREATE USER DBCLAW_MON
IDENTIFIED BY "<强密码>"
DEFAULT TABLESPACE USERS
TEMPORARY TABLESPACE TEMP
ACCOUNT UNLOCK;

GRANT CREATE SESSION TO DBCLAW_MON;
GRANT SELECT_CATALOG_ROLE TO DBCLAW_MON;
GRANT SELECT ANY TABLE TO DBCLAW_MON;
GRANT EXECUTE ON SYS.DBMS_XPLAN TO DBCLAW_MON;
```

### 可选增强

如需终止会话：

```sql
GRANT ALTER SYSTEM TO DBCLAW_MON;
```

### 一句话说明

Oracle 用系统只读角色加全表只读权限，是最好解释、最容易审批的做法。

## SAP HANA

### 推荐说明

- 最简单方案：`MONITORING + CATALOG READ + 业务 schema SELECT`
- 建议通过自定义角色统一封装，便于审批与后续维护

### 最简脚本

```sql
CREATE USER DBCLAW_MON PASSWORD "<强密码>" NO FORCE_FIRST_PASSWORD_CHANGE;

CREATE ROLE DBCLAW_MONITOR_ROLE;

GRANT MONITORING TO DBCLAW_MONITOR_ROLE;
GRANT CATALOG READ TO DBCLAW_MONITOR_ROLE;

GRANT SELECT ON SCHEMA "APP_SCHEMA" TO DBCLAW_MONITOR_ROLE;
GRANT SELECT ON SCHEMA "APP_SCHEMA2" TO DBCLAW_MONITOR_ROLE;

GRANT DBCLAW_MONITOR_ROLE TO DBCLAW_MON;
```

### 可选增强

如需终止会话：

```sql
GRANT SESSION ADMIN TO DBCLAW_MON;
```

### 一句话说明

HANA 建议用统一只读监控角色封装所有授权，最方便后续审计与复用。

## 四、推荐客户确认口径

建议对客户统一说明如下：

- DBClaw 默认使用专用只读监控账号接入数据库
- 该账号可以读取业务数据、元数据和系统监控数据
- 该账号默认不具备写权限
- 该账号默认不具备终止会话权限
- 若客户需要更强运维能力，可按需追加增强权限

## 五、备注

- 各数据库不同版本在细节语法和预定义角色上可能存在差异，若现场数据库版本较老，应由 DBA 按本版本语法微调。
- PostgreSQL / openGauss 的 `pg_stat_statements`、MySQL 的 `performance_schema` 等扩展或功能需数据库侧已启用。
- Oracle 建议在目标 PDB 中创建监控账号，不建议直接在 CDB Root 中混用业务监控权限。
