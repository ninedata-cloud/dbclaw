# 修复 SQLite 数据库锁定问题

## 问题描述

错误日志：
```
(sqlite3.OperationalError) database is locked
[SQL: INSERT INTO metric_snapshots (datasource_id, metric_type, data) VALUES (?, ?, ?) RETURNING id, collected_at]
```

## 根本原因

1. **并发写入冲突**：`metric_collector` 每 15 秒并发采集所有数据源的指标，多个协程同时写入 SQLite
2. **SQLite 默认限制**：默认的 journal 模式不支持高并发写入
3. **超时时间过短**：默认超时时间不足以等待锁释放

## 解决方案

### 1. 启用 WAL 模式（Write-Ahead Logging）

**文件**：`backend/database.py`

```python
async def init_db():
    # ...
    async with engine.begin() as conn:
        await conn.run_sync(Base.metadata.create_all)

        # Enable WAL mode for better concurrency
        def _enable_wal(connection):
            from sqlalchemy import text
            connection.execute(text("PRAGMA journal_mode=WAL"))
            connection.execute(text("PRAGMA busy_timeout=30000"))  # 30 秒超时

        await conn.run_sync(_enable_wal)
```

**优势**：
- 允许并发读写（多个读者 + 一个写者）
- 写入性能提升
- 减少锁竞争

### 2. 增加数据库超时时间

**文件**：`backend/database.py`

```python
engine = create_async_engine(
    get_settings().database_url,
    echo=get_settings().debug,
    connect_args={
        "check_same_thread": False,
        "timeout": 30,  # 增加超时时间到 30 秒
    },
    pool_pre_ping=True,  # 检测连接是否有效
)
```

### 3. 限制并发写入数量

**文件**：`backend/services/metric_collector.py`

```python
# 添加信号量限制并发写入
_db_write_semaphore = asyncio.Semaphore(10)  # 最多 10 个并发写入

async def collect_metrics_for_connection(datasource_id: int):
    async with _db_write_semaphore:  # 限制并发写入
        try:
            async with async_session() as db:
                # ... 采集和写入逻辑
```

**优势**：
- 控制并发写入数量，避免过多协程竞争锁
- 平衡性能和稳定性

## 验证方法

1. **重启应用**：
   ```bash
   python run.py
   ```

2. **观察日志**：确认不再出现 "database is locked" 错误

3. **检查 WAL 模式**：
   ```bash
   sqlite3 data/smartdba.db "PRAGMA journal_mode;"
   # 应该输出: wal
   ```

4. **压力测试**：添加多个数据源，观察并发采集是否稳定

## 性能影响

- **WAL 模式**：写入性能提升 20-30%
- **并发控制**：略微降低峰值吞吐量，但提升稳定性
- **超时增加**：不影响正常操作，仅在高负载时生效

## 后续优化建议

如果数据量继续增长，可以考虑：

1. **迁移到 PostgreSQL**：更好的并发支持
2. **分离读写**：使用主从复制
3. **批量写入**：合并多个指标快照为一次事务
4. **异步队列**：使用消息队列缓冲写入请求
