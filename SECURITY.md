## 报告漏洞

我们非常重视 DBClaw 的安全性。如果您发现了安全漏洞，请负责任地向我们披露。

### 报告流程

1. **请勿公开披露**：在我们有机会修复之前，请不要公开披露漏洞
2. **发送报告**：请通过以下方式之一报告安全问题：
   - 发送邮件至：security@ninedata.com（推荐）
   - 在 GitHub 上创建私密安全公告
3. **提供详细信息**：请在报告中包含：
   - 漏洞描述
   - 重现步骤
   - 影响范围
   - 可能的修复建议（如果有）
   - 您的联系方式

### 响应时间

- **初步响应**：我们将在 48 小时内确认收到您的报告
- **评估时间**：我们将在 7 个工作日内评估漏洞的严重性
- **修复时间**：
  - 严重漏洞：14 天内发布补丁
  - 高危漏洞：30 天内发布补丁
  - 中低危漏洞：60 天内发布补丁

### 致谢

我们感谢所有负责任地报告安全问题的研究人员。在修复发布后，我们将在 CHANGELOG.md 中公开致谢（除非您希望保持匿名）。

## 安全最佳实践

### 部署建议

1. **加密密钥管理**
   - 生产环境必须使用强随机的 `ENCRYPTION_KEY`
   - 定期轮换加密密钥
   - 使用密钥管理服务（如 AWS KMS、HashiCorp Vault）

2. **密码策略**
   - 修改默认管理员密码 `admin1234`
   - 使用强密码（至少 12 位，包含大小写字母、数字和特殊字符）
   - 启用密码过期策略
   - 内置 CLI（`scripts/reset_admin_password.py`）可在持有服务器或容器访问权限且能读取 `.env` 或 `runtime.env` 时重置登录密码；请严控相关文件与宿主机权限，避免未授权恢复

3. **网络安全**
   - 使用 HTTPS/TLS 加密传输
   - 限制管理端口访问（使用防火墙或安全组）
   - 配置反向代理（Nginx、Caddy）

4. **数据库安全**
   - 元数据库使用独立的强密码
   - 限制数据库网络访问
   - 定期备份元数据库

5. **容器安全**
   - 使用非 root 用户运行容器
   - 定期更新基础镜像
   - 扫描镜像漏洞

6. **日志和监控**
   - 启用访问日志
   - 监控异常登录行为
   - 定期审计用户权限

### 已知安全限制

1. **技能执行沙箱**
   - 技能代码使用受限的 Python 沙箱执行
   - 禁用了 `__builtins__`，但仍可能存在逃逸风险
   - 建议只使用内置技能或经过审计的自定义技能

2. **SQL 注入防护**
   - 大部分查询使用参数化查询
   - 动态 SQL 构建需要谨慎审查

3. **XSS 防护**
   - 前端使用 `_escapeHtml` 方法转义用户输入
   - 避免直接使用 `innerHTML` 插入未转义内容

4. **CSRF 防护**
   - 使用 SameSite Cookie 策略
   - 建议在生产环境启用 CSRF Token

## 合规性

DBClaw 设计时考虑了以下合规要求：

- **数据加密**：敏感数据（密码、密钥）使用 Fernet 加密存储
- **访问控制**：基于角色的访问控制（RBAC）
- **审计日志**：记录关键操作日志
- **数据隔离**：多租户数据隔离
