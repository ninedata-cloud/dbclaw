# 资源大盘健康状态功能

## 功能概述

资源大盘现在可以实时展示每个数据源的健康状态，判断标准与异常诊断触发标准完全一致。

## 实现内容

### 后端实现

#### 1. 新增API端点 (`backend/routers/metrics.py`)

**端点**: `GET /api/metrics/{conn_id}/health`

**功能**: 检查数据源健康状态

**返回格式**:
```json
{
  "healthy": true/false,
  "status": "healthy"/"warning"/"critical"/"unknown",
  "violations": [...],
  "message": "状态描述"
}
```

**判断逻辑**:

1. **数据可用性检查**
   - 无监控数据 → `unknown`
   - 监控数据过期（>5分钟）→ `unknown`

2. **阈值规则检查**（基于巡检配置）
   - 未配置阈值 → `healthy`（正常，未配置阈值）
   - 配置了简单阈值规则 → 检查每个指标是否超过阈值
   - 配置了自定义表达式 → 评估表达式是否为真

3. **状态判定**
   - 无违规 → `healthy`（正常）
   - 有违规 → `critical`（异常）

#### 2. 支持的阈值规则

**简单阈值规则**:
```json
{
  "cpu_usage": {"threshold": 80, "duration": 60},
  "memory_usage": {"threshold": 85, "duration": 60},
  "disk_usage": {"threshold": 90, "duration": 300},
  "connections": {"threshold": 100, "duration": 120}
}
```

**自定义表达式**:
```json
{
  "custom_expression": {
    "expression": "cpu_usage > 50 and connections > 20",
    "duration": 60
  }
}
```

#### 3. 指标名称映射

支持多种指标名称变体：
- `cpu_usage` → `cpu_usage_percent`, `cpu_percent`
- `memory_usage` → `memory_usage_percent`, `mem_percent`
- `disk_usage` → `disk_usage_percent`, `disk_percent`
- `connections` → `connections_active`, `active_connections`, `connection_count`, `threads_connected`

### 前端实现

#### 1. API客户端更新 (`frontend/js/api.js`)

新增方法:
```javascript
getDatasourceHealth(connId) { return this.get(`/api/metrics/${connId}/health`); }
```

#### 2. Dashboard页面更新 (`frontend/js/pages/dashboard.js`)

**显示变化**:
- 原来: "状态" 字段固定显示 "在线"
- 现在: "健康状态" 字段动态显示健康/异常状态

**状态展示**:
- ✓ 健康 (绿色) - 所有指标正常
- ✗ 异常 (红色) - 检测到指标超过阈值
- ⚠ 警告 (橙色) - 预留用于未来的警告级别
- 未知 (灰色) - 无数据或数据过期

**刷新机制**:
- 自动刷新: 每15秒更新一次
- 手动刷新: 点击"刷新"按钮

#### 3. CSS样式 (`frontend/css/main.css`)

新增 `.health-status` 样式类，用于美化健康状态显示。

## 使用场景

### 场景1: 正常运行的数据源

**配置**: CPU阈值80%，持续60秒
**当前状态**: CPU 45%
**显示**: ✓ 健康 (绿色)

### 场景2: CPU超载的数据源

**配置**: CPU阈值80%，持续60秒
**当前状态**: CPU 95%
**显示**: ✗ 异常 (红色)
**提示**: "检测到 1 个指标异常"

### 场景3: 多指标异常

**配置**: 
- CPU阈值80%
- 内存阈值85%
- 连接数阈值100

**当前状态**: 
- CPU 92%
- 内存 90%
- 连接数 120

**显示**: ✗ 异常 (红色)
**提示**: "检测到 3 个指标异常"

### 场景4: 自定义表达式

**配置**: `cpu_usage > 50 and connections > 20`
**当前状态**: CPU 65%, 连接数 35
**显示**: ✗ 异常 (红色)

### 场景5: 未配置阈值

**配置**: 无阈值规则
**显示**: ✓ 健康 (绿色)
**提示**: "正常（未配置阈值）"

### 场景6: 监控数据缺失

**情况**: 超过5分钟未收到监控数据
**显示**: 未知 (灰色)
**提示**: "监控数据过期 (X分钟前)"

## 与异常诊断的关系

健康状态检查使用的判断逻辑与异常诊断触发标准**完全一致**：

1. **相同的阈值配置**: 都读取 `inspection_configs` 表的 `threshold_rules`
2. **相同的判断逻辑**: 都使用相同的指标提取和阈值比较逻辑
3. **不同的触发条件**:
   - 健康状态: 实时检查当前指标值
   - 异常诊断: 需要持续超过配置的 `duration` 时间才触发

**示例**:
- 配置: CPU > 80%, 持续60秒触发诊断
- 当前: CPU 95%
- 健康状态: 立即显示"异常"
- 异常诊断: 需要CPU持续95%超过60秒才触发

## 配置方法

在数据源列表页面，点击数据源的"巡检配置"按钮，可以配置阈值规则：

1. **预设阈值**: 勾选并设置CPU、磁盘、连接数等阈值
2. **自定义表达式**: 使用Python表达式定义复杂的异常条件

配置保存后，资源大盘会立即使用新的阈值规则进行健康检查。

## 技术细节

### 性能优化

- 健康检查与指标获取并行执行
- 使用最新的监控快照，无需额外查询历史数据
- 前端缓存健康状态，避免频繁请求

### 错误处理

- API调用失败时，保持显示为 "--"
- 表达式评估失败时，视为不违规
- 指标缺失时，跳过该指标的检查

### 扩展性

- 支持添加新的指标名称映射
- 支持添加新的状态级别（如 warning）
- 支持自定义健康检查规则
