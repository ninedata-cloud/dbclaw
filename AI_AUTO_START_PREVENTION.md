# AI自动启动数据库服务的防护措施

## 问题描述

在连接失败诊断过程中，AI可能会自动执行启动数据库服务的命令（如 `systemctl start`、`docker start` 等），导致未经用户授权的系统状态变更。

## 根本原因

1. **连接失败检测链路**：
   - `metric_collector.py` 定时检测连接失败
   - 触发 `connection_failure` 类型的AI诊断
   - AI使用 `CONNECTION_FAILURE_DIAGNOSIS_PROMPT` 进行诊断

2. **危险技能可用**：
   - `execute_any_os_command.yaml` 技能允许执行任何OS命令
   - AI在诊断过程中可以调用此技能
   - 没有明确的限制禁止AI自动启动服务

## 解决方案

### 修改的文件

**backend/agent/prompts.py**

在三个系统提示词中添加了明确的操作限制：

#### 1. DIAGNOSTIC_PROMPT（诊断模式）

```python
**🚨 操作限制（最高优先级）：**
- **禁止执行任何启动、重启、停止服务的命令**
- **禁止执行任何修改系统配置、数据库配置的命令**
- **禁止执行任何可能影响系统状态的写操作**
- **你只负责诊断和提供建议，不负责自动修复**
- 在建议中提供修复命令时，必须明确标注"需要人工执行"
```

#### 2. ADMINISTRATIVE_PROMPT（管理模式）

```python
**🚨 操作限制（最高优先级）：**
- **禁止执行任何启动、重启、停止服务的命令**
- **禁止执行任何修改系统配置、数据库配置的命令**
- **禁止执行任何可能影响系统状态的写操作**
- **你只负责执行用户明确授权的操作**
- 对于可能影响系统的操作，必须先向用户确认，并由人工决策执行
```

#### 3. CONNECTION_FAILURE_DIAGNOSIS_PROMPT（连接失败诊断模式）

```python
**🚨 严格禁止的操作（最高优先级）：**
- **禁止执行任何启动数据库服务的命令**（如 systemctl start、service start、docker start 等）
- **禁止执行任何重启服务的命令**（如 systemctl restart、service restart、docker restart 等）
- **禁止执行任何修改系统配置的命令**（如修改防火墙规则、修改数据库配置文件等）
- **禁止执行任何可能影响系统状态的写操作**
- **你的职责仅限于诊断和提供建议**，不包括自动修复
- 在解决方案中，你可以提供修复命令的示例，但**必须明确标注"需要人工执行"**
- 所有涉及启动、重启、修改配置的操作，都必须在报告中说明"请管理员手动执行以下命令"

**诊断原则：**
- **只读诊断**：只执行查询、查看、检查类的命令
- **不自动修复**：发现问题后，在报告中提供解决方案，但不自动执行修复操作
- **人工决策**：所有可能影响系统的操作都需要人工审核和执行
```

## 防护机制

### 1. 提示词级别限制

- 在系统提示词中明确禁止AI执行启动、重启、修改配置等操作
- 使用醒目的标记（🚨）和加粗文本强调限制的重要性
- 标注为"最高优先级"，确保AI优先遵守这些规则

### 2. 职责明确

- AI的职责限定为：诊断问题、提供建议
- 不包括：自动修复、自动启动服务、修改配置
- 所有修复操作都需要人工审核和执行

### 3. 报告格式要求

- 在诊断报告中提供修复命令时，必须标注"需要人工执行"
- 明确说明"请管理员手动执行以下命令"
- 将AI定位为建议者，而非执行者

## 使用影响

### 对用户的影响

**之前**：
- AI可能自动启动停止的数据库服务
- 用户不知道系统状态被修改
- 可能导致意外的服务启动

**之后**：
- AI只进行诊断，不自动修复
- 在报告中提供修复建议和命令
- 用户需要手动执行修复命令
- 用户完全掌控系统状态变更

### 诊断报告示例

AI现在会生成类似这样的报告：

```markdown
## 六、解决方案

### 🔧 立即行动项

#### 方案 1：启动数据库服务

**问题**：数据库服务未运行

**解决步骤**（⚠️ 需要人工执行）：
1. 检查服务状态
2. 启动数据库服务
3. 验证服务是否正常运行

**命令示例**（请管理员手动执行以下命令）：
```bash
# 检查服务状态
systemctl status mysql

# 启动服务
sudo systemctl start mysql

# 验证服务
systemctl status mysql
```
```

## 其他防护层

虽然本次修改主要在提示词层面，但系统还有其他防护机制：

1. **技能权限系统**：`execute_any_os_command` 需要特定权限
2. **审计日志**：所有技能执行都有日志记录
3. **用户确认**：可以配置需要用户确认的操作

## 未来增强建议

1. **技能级别限制**：在 `execute_any_os_command.yaml` 中添加命令黑名单
2. **执行前确认**：对危险操作强制要求用户确认
3. **只读模式**：提供一个完全只读的诊断模式
4. **命令白名单**：只允许执行预定义的安全命令列表

通过在AI系统提示词中添加明确的操作限制，我们确保了：

1. ✅ AI不会自动启动、重启或停止数据库服务
2. ✅ AI不会自动修改系统或数据库配置
3. ✅ AI只负责诊断和提供建议
4. ✅ 所有修复操作都需要人工审核和执行
5. ✅ 用户完全掌控系统状态变更
