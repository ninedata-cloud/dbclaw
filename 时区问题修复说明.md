# 时区问题修复说明

## 问题描述

性能监控采集的数据时间晚了 8 小时，导致查询历史数据时出现时间偏差。

## 根本原因

1. **数据库默认值使用 UTC 时间**
   - `MetricSnapshot` 模型的 `collected_at` 字段使用了 `server_default=func.now()`
   - SQLite 的 `CURRENT_TIMESTAMP` 返回 UTC 时间
   - 中国时区为 UTC+8，导致时间晚了 8 小时

2. **代码未显式设置时间**
   - 创建 `MetricSnapshot` 时没有显式传入 `collected_at` 参数
   - 依赖数据库默认值，导致使用了 UTC 时间

## 修复方案

### 1. 修改模型定义

**文件**: `backend/models/metric_snapshot.py`

```python
# 修改前
collected_at = Column(DateTime, server_default=func.now(), index=True)

# 修改后
collected_at = Column(DateTime, nullable=False, index=True)  # 移除 server_default
```

### 2. 显式设置本地时间

**文件**: `backend/services/metric_collector.py`

```python
# 修改前
snapshot = MetricSnapshot(
    datasource_id=datasource_id,
    metric_type="db_status",
    data=normalized_status,
)

# 修改后
snapshot = MetricSnapshot(
    datasource_id=datasource_id,
    metric_type="db_status",
    data=normalized_status,
    collected_at=now(),  # 使用本地时间
)
```

### 3. 修复历史数据

**迁移脚本**: `backend/migrations/fix_metric_snapshot_timezone.py`

```sql
-- 将所有历史记录的时间加 8 小时
UPDATE metric_snapshots
SET collected_at = datetime(collected_at, '+8 hours')
```

## 预防措施

1. **统一使用 `datetime_helper.now()`**
   - 所有需要当前时间的地方使用 `now()` 函数
   - 避免使用 `datetime.utcnow()` 或数据库默认值

2. **显式设置时间字段**
   - 创建记录时显式传入时间参数
   - 不依赖数据库的 `server_default`

3. **时区一致性**
   - 整个系统统一使用本地时间（无时区信息）
   - SQLite 不支持时区，使用 naive datetime
